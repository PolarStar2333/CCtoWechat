1. 工作原理

微信 ⟷ 腾讯 iLink 服务器 ⟷ bridge.py ⟷ Claude Code 终端
环节实现
微信 ⇄ 电脑腾讯 iLink Bot API（2025 年开放的官方接口），HTTP JSON 长轮询，无需公网 IP
消息注入终端Windows keybd_event API 模拟 Ctrl+V 粘贴 + 回车
抓取 Claude 回复监控 Claude Code JSONL 对话日志，等 end_turn 后收集整轮所有文字
这套方案完全独立于 OpenClaw/CowAgent 等框架，直接调用腾讯底层 API，无需安装任何微信插件或 npm 包。

3. 部署步骤

第一步：准备文件

将以下文件放到一个目录中（例 D:\CCtoWechat\）：

  • bridge.py — 桥接主程序
  • run.bat — Windows 启动脚本

第二步：启动桥接

cd D:\CCtoWechat
python bridge.py

或直接双击 run.bat。

首次启动会打印一个二维码 URL：

请用微信扫描：
  https://liteapp.weixin.qq.com/q/7GiQu1?qrcode=...

第三步：微信扫码绑定

把二维码 URL 发到微信里点开，按提示完成授权。

终端显示 登录成功 (xxxx@im.bot) 即完成绑定。

凭证自动保存到 credentials.json，以后启动无需重新扫码。该文件包含你的 Bot Token，请勿分享给他人。

第四步：开始对话

看到 监听中... 后，在微信里发给刚才绑定的 Bot，消息会自动注入终端，Claude 的回复也会自动发回微信。

5. 从终端主动给微信发消息

桥接启动后在 127.0.0.1:9876 提供 HTTP /send 端点：

# curl
curl -X POST http://127.0.0.1:9876/send \
  -H "Content-Type: application/json" \
  -d '{"text": "你好，来自终端。"}'

# 或 Python
python -c "
import urllib.request, json
data = json.dumps({'text': '你好'}).encode()
urllib.request.urlopen(urllib.request.Request(
    'http://127.0.0.1:9876/send', data=data,
    headers={'Content-Type': 'application/json'}, method='POST'))
"
注意：需要你之前在微信发过至少一条消息，桥接才有你的 context_token。

6. 远程权限审核

当你离开电脑、只带手机时，Claude Code 的权限弹窗谁来处理？CCtoWechat 支持远程审核：

工作流

Claude 弹权限 → Hook 写文件 → 桥接检测 → 微信通知你 → 你回 /yes 或 /no → 桥接注入终端

配置

在 Claude Code 的 ~/.claude/settings.json 中添加：

{
  "hooks": {
    "PermissionRequest": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "python 你的路径/approval_hook.py"
      }]
    }]
  }
}

使用

  1. Claude 执行需要权限的操作 → Hook 自动触发
  2. 微信收到通知：工具: Bash\n内容: rm -rf ...\n\n回复 /yes 批准，/no 拒绝
  3. 你回复 /yes → 桥接注入 yes 到终端 → Claude 继续执行
  4. 你回复 /no → 拒绝

去重机制

Claude Code 的 TUI 刷新会导致同一权限弹窗重复触发 Hook。桥接对比待发内容和上一条通知——一样就跳过，不一样才发。同一弹窗只骚扰一次。

审核消息以 /yes 或 /no 开头才会被识别为审核回复。普通消息不受影响。

9. 技术架构

8.1 iLink Bot API

腾讯 2025 年推出的微信个人 Bot 通信协议，域名 ilinkai.weixin.qq.com。纯 HTTP JSON，7 个端点：

端点方法用途
/ilink/bot/get_bot_qrcodeGET获取登录二维码
/ilink/bot/get_qrcode_statusGET轮询扫码状态
/ilink/bot/getupdatesPOST长轮询收消息（hold 约 35s）
/ilink/bot/sendmessagePOST发送回复（需 context_token）
/ilink/bot/getuploadurlPOST获取 CDN 上传签名
/ilink/bot/getconfigPOST获取 typing ticket
/ilink/bot/sendtypingPOST发送"正在输入"状态
关键设计：每条消息都带 context_token，回复时必须原样携带，否则消息无法投递到正确的对话窗口。

8.2 回复抓取策略

bridge.py 监控 Claude Code 的 JSONL 会话文件（路径：~/.claude/projects/<项目>/<session>.jsonl），采用精确的事件驱动策略：

  1. 第一阶段：等待注入的用户消息出现在 JSONL 中
  2. 第二阶段：等待 stop_reason = "end_turn" 的 assistant 消息（过滤掉思考过程和工具调用）
  3. 拼接：回头收集本轮对话中所有 assistant 文本（包括 tool_use 前的上下文文字），用空行拼接后一并发送

这个策略不需要任何时间猜测，完全由 Claude Code 的原生事件驱动。

8.3 终端注入

经过多次方案对比（PowerShell SendKeys / pyautogui.write / ctypes SendInput），最终采用：

  • pyperclip.copy() 将消息写入剪贴板
  • Windows keybd_event API 模拟 Ctrl+V + Enter

这是最底层的按键模拟方案，经过后台进程注入验证，稳定性最好。

8.4 会话自动跟随

不加 --session 时，bridge.py 在每条消息处理前重新扫描 ~/.claude/projects/，找最近修改的 JSONL。这保证了切换到新的 Claude Code 会话时无需重启。

10. 优势与局限

优势

优势说明
🔑 零 API Key不需要 Anthropic / OpenAI / 中转站 API Key。直接调用本机 Claude Code CLI
🌐 零公网 IPiLink 长轮询拉取消息，不需要服务器、不需要 frp/ngrok
📦 零框架依赖不依赖 OpenClaw、CowAgent、Node.js 等重型框架，只依赖 httpx
👁 代码完全可审计核心仅 ~370 行 Python，30 分钟即可通读
🛡 无 Agent 安全风险不做文件系统访问、不执行命令、不运行 Agent。Claude 的回复是你自己在终端看到的，不会偷偷操作
🏛 腾讯官方通道iLink Bot API 是腾讯 2025 年开放的个人微信 Bot 协议，非逆向/注入，无封号风险
🔄 自动跟随会话切换 Claude Code 终端时自动检测最新 JSONL，无需重启
📝 完整回复拼接等 end_turn 后回溯整轮所有文字，工具调用前的上下文也不会丢
♾ 无超时等待阶段 2 等 end_turn 无时间上限，Claude 思考再久也不会丢回复
📡 HTTP /send 端点内置 HTTP 接口，可在脚本、定时任务中主动向微信推送消息

局限

局限说明影响程度
🪟 仅 Windows终端注入依赖 keybd_event API 和剪贴板，macOS/Linux 待适配高
📝 仅文本不支持图片、语音、文件收发（iLink 协议支持，待实现）中
⏳ 无流式输出回复在 Claude 完成后一次性发送，微信里看不到逐字输出的效果中
🔞 不支持群聊iLink 协议限制，Bot 只能私聊中
🕐 思考期间静默Claude 思考过程中微信无反馈。如思考超过 10 分钟，阶段 1 可能超时中
💻 终端须可见终端窗口最小化时注入可能失败（Windows 焦点机制）低
⚠ iLink 协议实验性腾讯未正式文档化该 API，可能在未来版本中变更中
⏰ 24 小时会话窗口iLink 的 Bot Token 有效期约 24 小时，过期需重新扫码低
🔤 中文编码终端日志中中文显示为乱码（GBK vs UTF-8，不影响实际功能）低

11. 故障排除

Q: 消息没注入到终端
A: 1) 确保终端窗口在桌面可见（未最小化）；2) pip install pyperclip；3) 重启桥接。
Q: 微信收不到回复
A: 1) 确保你先在微信发过至少一条消息（桥接需要 context_token）；2) 确认只开了一个 Claude Code 终端；3) 等 Claude 完成整轮思考后再发下一条消息。
Q: 切换 Claude Code 会话后收不到回复
A: 不加 --session 参数，桥接会自动跟随最新会话。如果已经用了 --session，删掉参数重启即可。
Q: Session 过期（错误码 -14）
A: 删除 credentials.json 和 cursor.json，重新运行扫码授权。
Q: 二维码过期
A: 程序会自动刷新（最多 3 次）。多次过期的话重启桥接即可。
Q: 回复只收到后半段
A: 已修复——v4 会等 end_turn 后收集整轮所有 assistant 文本再发送。如果仍有此问题，请提交 Issue。
Q: 端口 9876 被占用
A: 有其他程序占用了该端口（可能是上次未正常退出的桥接进程）。在任务管理器中找到占用该端口的 python.exe 进程并结束。
